## Session 10 ŌĆö Cloudflare proxy rollout, DNS-01 cert issuance

### What we worked on

A corporate laptop couldn't reach `marie.lostriver.llc` ŌĆö "site can't be reached." Diagnosed as corp SSL-inspection proxy sending TCP RST during TLS handshake to DigitalOcean IPs (origin 174.138.33.106). Fixed by putting all `lostriver.llc` subdomains behind Cloudflare's orange-cloud proxy. Also moved Traefik cert issuance to DNS-01 via the Cloudflare API so certs keep renewing correctly behind the proxy and are decoupled from inbound port 80.

### Root cause

From the corp laptop:
- DNS resolved fine (corp resolver returned `174.138.33.106`)
- `nc -vz marie.lostriver.llc 443` succeeded (TCP reached origin)
- `curl -v` died mid-TLS with `SSL_ERROR_SYSCALL` right after Client Hello
- Same error with a bogus SNI directed at the IP ŌåÆ rules out domain-specific SNI filtering
- Classic signature of SSL-inspection proxy rejecting connections based on destination IP category (IaaS / uncategorized)

Server was healthy the whole time (external curl: HTTP 200, valid LE cert).

### Decisions made

- **Keep both resolvers during transition**: acme-dns.json is currently empty because Traefik is still serving valid HTTP-01 certs from acme.json (expiring 2026-07-14). The switch to DNS-01 is prospective ŌĆö next renewal will exercise the path. Safer than forcing re-issuance now.
- **No CF cache rule for `/api/*`**: verified all 3 API domains return `cf-cache-status: DYNAMIC` by default (CF's built-in logic doesn't cache non-static extensions with no cache headers). Revisit if caching issues appear.
- **DNS-01 over HTTP-01** because: (a) survives origin port-80 restrictions, (b) works cleanly with CF proxy, (c) prereq for future "firewall :443 to CF IPs only" hardening.
- **Orange cloud is now the default convention** for new projects. Documented in README + init-project.sh.

### Issues encountered

- **CF API token scope mismatch**: scoped token can flip proxy (DNS:Edit) but can't change Zone Settings (SSL mode, Always-HTTPS) or add Cache Rules. Left as dashboard actions rather than broadening the token.
- **Local DNS cache masking verification**: first curl to `marie.lostriver.llc` after flipping orange cloud still hit origin IP. Had to force fresh lookup via `dig @1.1.1.1`. Not a bug, just a cache gotcha.
- **Stale `vagent-tunnel.lostriver.llc` entry in acme.json**: Traefik keeps trying to renew it, fails with NXDOMAIN. Harmless noise. Will clean up when moving away from HTTP-01 resolver entirely.
- **Stale caitie-api file-provider healthcheck**: Traefik logs spam about `caitie-api:3000` health check failing (container's been gone for days). Remove from `dynamic/dynamic.yml` in a follow-up.

- All settings env-driven: `VAULTWARDEN_HOST`, `_DOMAIN`, `_CERTRESOLVER`, `_ADMIN_TOKEN`, `_SIGNUPS`, `_VOLUME` (the last to reuse `vault_vaultwarden_data` so existing data + admin token survived).

### Gotchas

- **Gitignored `.env` is still `.env` after `git init` + `reset --hard`**: the repo's `.gitignore` takes effect immediately, so the server's existing `.env` stays untouched as an untracked file. No secret exposure risk.
- **Vaultwarden router not instantly visible**: first `/admin` hit via Traefik returned 404 because Traefik's Docker provider hadn't picked up the new container yet. Resolved in ~1-2s. Worth remembering for future migrations ŌĆö verify router via `/api/http/routers/<name>@docker` before declaring success.
- **`git checkout -B main origin/main` after `git init`** doesn't populate the working tree (the branch is created but files aren't materialized). Use `git reset --hard origin/main` instead to actually check files out.
