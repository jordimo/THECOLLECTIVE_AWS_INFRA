# Dev Diary — 2026-04-20

---

## Session 15 — aws01 overnight OOM / livelock; swap + mem_limits added

### What happened

aws01 (t3.medium, 10.251.8.172) became unreachable overnight and came back up this morning at 12:25 UTC. Ran the host through `journalctl -b -1` and reconstructed the failure.

**Timeline (UTC):**
- `2026-04-13 19:58` — last clean boot. Host up for 6d 8h before the incident.
- `2026-04-20 03:53:16` — kernel OOM killer invoked (`gfp_mask=0x140cca`, global_oom, mems_allowed=0). First victim: `langfuse-web`'s Next.js server (PID 3992666, UID 1001, `total-vm:1731948kB, anon-rss:593236kB`). The kernel `comm` field is limited to 15 chars so the OOM log reads `next-server (v1…` — confirmed against the running container: `docker exec langfuse-web ps` shows `next-server (v16.2.3)` as UID 1001(nextjs). Umami is also Next.js under UID 1001 but its RSS (182 MB) rules it out as the 593 MB victim.
- `03:53 → 05:02` — sustained memory pressure. journald "Under memory pressure, flushing caches." every 30–90s. snapd watchdog fired at `04:00:53`, systemd SIGKILLed snapd, snapd restart-loop counter climbed to 11.
- `05:02:38` — last journal entry of boot `-1`. Kernel effectively livelocked — no panic (`panic=-1` would have triggered an immediate reboot otherwise).
- `~12:25:55` — host boots back up. Likely manual or AWS auto-recovery; the 7-hour gap rules out kernel panic.

### Root cause

Undersized host running a too-dense workload, with zero swap to absorb allocation spikes.

- **t3.medium = 4 GB RAM, 2 vCPU.** No swap configured.
- **11 containers** on the host: `traefik`, `postgres`, `redis`, `langfuse-web`, `langfuse-worker`, `clickhouse`, `minio`, `umami`, `marie-api`, `marie-web`, `mainsite-web`.
- **Idle steady-state: 3.1 GB / 3.7 GB used (≈83%).** Langfuse alone = ~1.34 GB (web 568M + worker 408M + clickhouse 362M).
- When one process (Next.js, ~600 MB RSS) spiked, the kernel had ~600 MB free + buff/cache to reclaim. It OOM-killed the biggest cgroup, but that freed well under 1 GB while the rest of the system was still fighting for memory. With no swap and no slack anywhere else, the kernel fell into thrash: repeated cache flushes, systemd units hitting watchdog timeouts, and eventually the scheduler couldn't make forward progress. A single OOM kill cascaded into a full-host livelock.

### What got done

**1. 2 GB swapfile on aws01.**

```
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo "/swapfile none swap sw 0 0" | sudo tee -a /etc/fstab
```

Verified: `swapon --show` → `/swapfile file 2G 0B -2`. `free -h` → `Swap: 2.0Gi`. Persisted across reboot via `/etc/fstab`. No swappiness tweak — Ubuntu default `vm.swappiness=60` is fine; the point is to have a release valve, not to page aggressively.

**2. `mem_limit` on the three heavy services** in `docker-compose.yml` (unified infra compose, applies to local + DO + aws01):

- `langfuse-web`: `768m` (idle 568M; ~35% headroom)
- `langfuse-worker`: `512m` (idle 408M)
- `clickhouse`: `512m` (idle 362M)

With hard caps, a future memory spike OOM-kills the offending container instead of the random process with the highest `oom_score_adj` on the host. Docker's restart policy (`unless-stopped`) brings it back within seconds; the host stays alive.

`docker compose config --quiet` passes locally.

**NOT yet deployed** — the edit is in the repo but the aws01 containers are still running the old unlimited config. User needs to commit, push, `git pull` on aws01 + isidora, and `docker compose up -d` to recreate the three containers with the new limits. Langfuse ingestion pauses for a few seconds during recreate.

### Decisions

- **Swap over resize.** Resizing to t3.large (8 GB) would have been the cleaner fix, but it requires an instance stop/start and someone to actually do it. Swap is free, reversible, and in place now. Revisit resize if swap usage climbs above ~500 MB sustained — that means we're paging hot pages, not just providing a safety valve.
- **Hard `mem_limit` not `mem_reservation`.** Soft limits don't prevent OOM; they just hint to the kernel. Hard caps are the point — when Langfuse worker has a bad day, it should die alone.
- **Apply limits in the shared compose, not an aws01 overlay.** The same memory pressure profile will show up on DO (isidora is also memory-constrained) and on anyone's laptop. One file, one set of limits.
- **Don't cap the full container set.** `mem_limit` on every service is busywork; the OOM audit identified three services that together account for ~70% of container memory. Cap those; leave postgres/redis/traefik/minio/umami/marie-* alone for now.

### Issues encountered

- **Initial mis-attribution of the OOM victim.** The OOM log's truncated `next-server (v1…` looked like it could have been `marie-web` or `mainsite-web` — neither of those projects uses Next.js, though. Two of the third-party stacks on this host are built on Next.js: `langfuse-web` and `umami`. Comparing RSS (langfuse-web 568 MB idle vs. umami 182 MB) against the 593 MB kill makes it unambiguous: `langfuse-web`. Lesson: when a kernel `comm` field is truncated, `docker exec <container> ps` on every likely candidate resolves it in one command.
- **"The leak" is a misframing.** Langfuse-web was killed at 593 MB RSS; freshly booted today it's already at 568 MB. That's a 25 MB difference — normal working-set variation, not a leak. Langfuse-web simply *is* a ~600 MB Next.js application. The OOM killer picked it because it was the biggest cgroup at the moment the system hit zero free memory, not because it had grown pathologically. The actual trigger (the allocation that pushed an 83%-full host over the edge) was not captured by the kernel — could have been a Postgres query, a cron job, kernel page cache pressure, or a Marie research worker allocation. We won't know from the journal alone.
- **The host had been up for 6 days before crashing.** A fresh t3.medium with this exact workload would have the same ~83% idle footprint. The crash wasn't a slow leak counting down to any specific day — it was a stochastic spike on a host that had no headroom. Six-day uptime is a coincidence, not a countdown.

### Open threads

1. **Deploy the mem_limits to aws01 + isidora.** Commit the infra change → push → pull on both hosts → `docker compose up -d`. Order: isidora first (public, more eyes on it), aws01 second.
2. **Instance resize to t3.large.** Recommended if any of (a) swap usage sustained >500 MB, (b) another OOM fires within 30 days, (c) we add another stack to this host. Stop/start window: ~2 minutes downtime.
3. **Identified the trigger: Ubuntu overnight maintenance.** In the 60s window before OOM, eight maintenance units all ran and deactivated between 03:53:21 and 03:53:38: `fstrim`, `logrotate`, `dpkg-db-backup`, `systemd-tmpfiles-clean`, `esm-cache` (consumed 45s CPU alone), `fwupd-refresh` (failed), `man-db`, `update-notifier-download`. These are `/etc/cron.daily` + systemd timers firing around 03:00–03:53 UTC. Each alone is modest; all of them concurrently on a host with ~600 MB free is enough to tip into global OOM. Classic Ubuntu pattern — the fix isn't to disable these (they're needed), it's to make sure there's headroom when they run. Swap + mem_limits handle this. Followup: add a lightweight `docker stats --no-stream` cron writing to a rolling log on the host, so next time we have pre-OOM deltas instead of guessing.
4. **ClickHouse internal memory tuning.** 512M is a coarse backstop. ClickHouse has its own `max_server_memory_usage` and `max_memory_usage` settings that control per-query limits; worth configuring if we ever see ClickHouse OOMKilled in logs.
5. **Basic host-level alerting.** This failure went unnoticed for ~7 hours. Even a single cron job that pings `/api/health` from outside and mails on failure would have caught it. Would live in infra, not in the app repos.
6. **Consider moving mainsite or Marie off aws01.** Three Next.js apps + Langfuse v3 + Postgres + Clickhouse + MinIO + Umami is a lot for 4 GB. Swap + caps buy runway; they don't change the long-term density problem.
